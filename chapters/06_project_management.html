Ⅵ. 프로젝트 관리

1. 관리 철학

프로젝트 관리는 일정만 관리하는 것이 아니라 claim/evidence, 보안, 익명평가, 분량, 변경관리까지 함께 관리하는 체계로 작성한다. PM은 서사와 우선순위를 조정하고, Compliance와 Data/AI의 veto를 반영해 최종 문안을 조정한다. 이 장은 “어떻게 끝까지 통제할 것인가”를 보여주는 장이다.

2. 조직과 역할

| 역할 | 책임 | 본문에서의 기능 |
| --- | --- | --- |
| 제안총괄 PM / Chief Editor | 구조, 일정, 분량, 최종 조정 | 전체 문안의 일관성 유지 |
| Compliance Lead | RFP 적합성, 익명성, 제출 형식 | 형식/익명성 veto |
| Solution Architect | 기술 구조, 방법론 설계 | 기술 spine 책임 |
| Data / AI Lead | 검증 프레임, 오차, 데이터 품질 | evidence admissibility |
| Airport Ops / Domain Lead | 공항별 운영환경, 예외 규칙 | 공항별 validation logic |
| Security / Privacy / IP Lead | 보안, 개인정보, IP, API 제약 | 보호조건 정의 |
| QA / Testing Lead | 테스트 계획, acceptance criteria | 검증 체계 책임 |
| Support / Handover Lead | 교육, 안정화, 하자보수 | 운영전환 책임 |

3. 일정과 산출물 관리

Stage 1~2.5는 이미 완료되어 있으므로, 본 장에서는 Stage 3~5의 운영 원칙을 본문 언어로 설명한다. Stage 3는 본문 작성, Stage 4는 발표자료 압축, Stage 4 Preflight는 분량/익명성/정합성 검증, Stage 5는 freeze와 최종 QA다. 산출물은 본문과 발표자료가 동일 ledger를 공유하도록 관리해야 한다.

4. 위험관리와 변경관리

핵심 리스크

  • 약한 claim의 공통 spine 유입
  • 공항별 validation 흐름 붕괴
  • 본문과 발표자료 불일치
  • 익명평가 누락
  • 페이지 예산 초과
  • 외부기관 협의 지연

대응 원칙

  • Stage 2.5 reject claim은 Stage 3/4로 이동하지 않는다.
  • 근거가 바뀌면 downstream drafting 전에 이유를 기록한다.
  • Compliance와 Data/AI veto는 해소되기 전 무시하지 않는다.
  • 페이지 예산과 익명성은 preflight에서 다시 잠근다.

5. 보고 체계

보고 체계는 일정 보고, 산출물 보고, 이슈 보고, 리스크 보고, 변경 요청 보고로 구성한다. 각 보고는 claim/evidence ledger와 연결되어야 하며, 본문·발표자료의 메시지 차이를 허용하지 않는다. (PMR-001~PMR-004)

6. RACI형 책임표

| 항목 | PM | Compliance | Data/AI | Architect | QA |
| --- | --- | --- | --- | --- | --- |
| 구조 | A | C | C | R | C |
| 익명성 | C | A | C | C | R |
| 증거 승인 | C | C | A | R | C |
| 일정 | A | C | C | C | R |
| 품질 게이트 | C | A | A | C | R |

7. 일정 관리의 구체화

본문에서는 일정 자체를 숫자로 길게 나열하기보다는, 작성-검토-수정-승인-패키징의 순환을 중심으로 설명하는 것이 효과적이다. 각 단계는 산출물 기준으로 종료 조건을 정의해야 하며, 종료 조건이 명확해야 다음 단계로 넘어갈 수 있다. 이 원칙은 특히 분량 통제와 익명평가 scrub에 유효하다.

8. 권장 도식

  • Project governance map
  • Issue escalation flow
  • Milestone/Gantt chart
  • Change control loop
  • Deliverable approval path

9. 본문에 넣을 수 있는 문장

  • “프로젝트 관리는 일정, 산출물, 품질, 보안, 익명성, 변경관리를 하나의 통제체계로 운영한다.”
  • “Stage 2.5에서 reject된 claim은 본문과 요약서로 이동하지 않는다.”
  • “모든 변경은 근거가 바뀐 이유와 함께 기록한다.”

10. 사업 수행 체계의 실제 운영

프로젝트 수행 체계는 문서상 조직도보다 실제 의사결정 흐름이 중요하다. PM은 전체 우선순위를 조정하고, Architect는 구조와 일관성을 점검하며, QA는 검증 항목을 통제하고, Compliance는 익명성과 형식을 관리한다. Data/AI는 증거의 적합성을 판단하고, Airport Ops는 공항별 현실성을 보정한다. 이 역할 분담이 명확해야만 Stage 3 이후 본문과 Stage 4 이후 발표자료가 같은 결론을 유지할 수 있다.

11. 변경관리와 커뮤니케이션

공공 제안서에서는 변경관리와 커뮤니케이션이 결과 품질을 좌우한다. 요구사항이나 근거가 바뀌면, 변경 이유, 영향 범위, 수정 대상, 승인자를 기록해야 한다. 이 기록이 없으면 본문과 발표자료 사이의 불일치가 생긴다. 따라서 변경관리 표에는 변경 요청 내용뿐 아니라, 어떤 claim을 수정했고 어떤 표와 도식이 영향을 받았는지도 함께 적는 것이 바람직하다. 커뮤니케이션 측면에서는 주간 점검, 이슈 공유, 승인 요청, 제출 전 최종 확인의 흐름이 있어야 한다.

12. 산출물 통제와 제출 준비

제출 단계의 산출물 통제는 내용만이 아니라 형식, 분량, 익명성, 파일명, 버전 관리까지 포함한다. 본문에서 가장 중요한 것은 “무엇이 어디에 있는가”를 분명히 하는 것이다. 따라서 최종 제출 전에는 장별 파일, 매핑표, 도식 목록, 표 목록, 변경 이력표가 함께 정리되어야 한다. 이 장은 관리 체계가 단순 행정 절차가 아니라 품질을 보장하는 메커니즘임을 보여준다.
