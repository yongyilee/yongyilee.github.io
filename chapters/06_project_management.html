Ⅵ. 프로젝트 관리 초안

1) 관리 철학

  • 프로젝트 관리는 일정만 관리하는 것이 아니라, claim/evidence, 보안, 익명평가, 분량, 변경관리까지 함께 관리하는 체계로 작성한다.
  • PM은 서사와 우선순위를 조정하고, Compliance와 Data/AI의 veto를 반영해 최종 문안을 조정한다.

2) 조직과 역할

  • 제안총괄 PM / Chief Editor: 구조, 일정, 분량, 최종 문안 조정
  • Compliance Lead: RFP 적합성, 익명성, 제출 형식, 분량 관리
  • Solution Architect: 기술 구조와 방법론 설계
  • Data / AI Lead: 검증 프레임, 오차, 데이터 품질, admissibility 판단
  • Airport Ops / Domain Lead: 공항별 운영환경과 예외 규칙 반영
  • Security / Privacy / IP Lead: 보안, 개인정보, IP, Open API 제약 관리
  • QA / Testing Lead: 검증 계획, acceptance criteria, red-team 질문 관리
  • Support / Handover Lead: 교육, 안정화, 하자보수, 보고체계

3) 일정과 산출물

  • Stage 1~2.5는 이미 완료되어 있으며, 본 장에서는 이를 전제로 Stage 3 이후 작성·검토 일정을 설명한다.
  • Stage 3는 본문 초안 작성, Stage 4는 발표자료 압축, Stage 4 Preflight는 분량/익명성/정합성 검증, Stage 5는 freeze와 최종 QA로 구성한다.
  • 산출물 관리는 본문과 발표자료가 동일한 ledger를 공유하도록 설계한다.

4) 위험관리와 변경관리

  • 대표 리스크는 약한 claim의 공통 spine 유입, 공항별 validation 흐름 붕괴, 본문과 발표자료의 불일치, 익명평가 누락이다.
  • 변경관리에서는 근거가 바뀌면 downstream drafting 전에 이유를 기록한다.
  • Stage 2.5에서 reject된 claim은 Stage 3/4로 이동하지 않는다.

5) 보고 및 품질 게이트

  • 일정·산출물·이슈는 주기적으로 점검한다.
  • 본문 150쪽 이하, 발표자료 30쪽 이하, 익명평가 scrub 완료를 관리 게이트로 둔다.
  • QA/Testing Lead는 claim/evidence coverage와 score-matrix coverage를 함께 점검한다.
