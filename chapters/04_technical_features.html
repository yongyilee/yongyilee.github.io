Ⅳ. 기술 및 기능

1. 기술 구현의 전반 구조

이 장은 실제 시스템이 어떤 기능으로 구성되고, 어떤 데이터와 인터페이스를 통해 동작하는지 설명한다. 기능은 단순 목록이 아니라, 수속단계 정의 → 실시간 산출 → 혼잡도 판단 → 예측 → API/대시보드/리포트/맞춤형 안내로 이어지는 흐름으로 서술한다. 기능별 설명은 반드시 RFP 기능 요구사항과 연결되며, 새로 만든 주장이나 수치는 Stage 2.5 승인 없이 사용하지 않는다.

2. 기능별 상세 서술

2.1 국제선 수속단계 정의 및 관리 (SFR-001)

국제선 수속단계는 공항별 운영 환경에 따라 단계와 예외 규칙이 다르므로, 공통 단계 프레임과 공항별 단계 정의를 분리해 관리한다. 제안서는 표준 단계 정의, 예외 단계 등록, 단계별 관리항목, 변경 이력 관리 방식을 함께 제시해야 한다. 이때 단계 정의는 단순 화면 문구가 아니라 데이터 스키마와 검증 기준을 함께 가진다.

2.2 단계별 실시간 소요시간 산출 (SFR-002)

실시간 소요시간은 체크인, 보안검색, 출국심사, 탑승구 이동 등 단계별로 산출되며, 데이터 지연·누락·이상치에 대비한 전처리와 대체 산정 규칙이 필수다. 산출 로직은 수집 → 정제 → 계산 → 배포의 4단계로 이해할 수 있으며, 현장 운영 변화가 있을 때 재학습 또는 재보정이 가능해야 한다.

2.3 단계별 혼잡도 산출 및 등급화 (SFR-003)

혼잡도는 단순 체감치가 아니라 운영자가 이해할 수 있는 등급 체계로 정의해야 한다. 기준값, 경보 기준, 단계별 표시 방식, 운영자 화면 반영 방식을 함께 적는다. 혼잡도 등급은 현장 조치와 연결되어야 하므로, 등급 변경 시 알림과 보고가 자동으로 이어지는 구조가 바람직하다.

2.4 AI 기반 소요시간 및 혼잡도 예측 (SFR-004)

예측 모델은 국제선 유형·환경별 데이터 테이블, 항공편 스케줄, 시간대, 요일, 예상 승객수, 기상, 공항별 운영특성을 반영해야 한다. 본문에서는 모델명보다 입력 특성, 검증 프레임, 오차 분석, 운영 적합성 평가에 초점을 둔다. 또한 예측값은 단독 진실값이 아니라 실측·검증과 함께 해석되는 운영 정보로 설명해야 한다.

2.5 API 개발 및 서비스 노출 (SFR-005)

API는 내부 운영 시스템과 외부 서비스가 공유할 수 있도록 설계하되, 제공 가능 항목, 인증, 권한, 상태코드, 에러 응답을 먼저 정의한다. 공개 범위를 과장하지 않고, 향후 확장 가능한 구조로 제시한다. 외부 서비스 연계는 기능 구현과 동시에 보안·개인정보·권한 정책과 함께 설명되어야 한다.

2.6 운영자용 대시보드 (SFR-006)

운영자 화면은 실시간 상태, 예측, 경보, 정합성 이슈, 공항별 예외를 동시에 볼 수 있어야 한다. 대시보드는 현장 운영자의 판단을 보조하는 도구로 서술한다. 즉, 자동화가 운영자를 대체하는 것이 아니라 운영 의사결정을 더 빠르고 정확하게 만드는 구조를 제시해야 한다.

2.7 통계 분석 및 리포트 (SFR-007)

리포트는 서비스 운영 현황, 오차 추이, 예외 발생률, 공항별 특이사항, 개선 과제를 정리하는 형태로 설계한다. 이 리포트는 주간·월간 운영 점검, 분기 검토, 개선 계획 수립에 연동되어야 한다.

2.8 이용객 맞춤형 안내 (SFR-008)

이용객 조건에 따라 수속 소요시간과 혼잡도 안내를 개인화할 수 있어야 한다. 다만 개인정보 활용과 비식별 기준은 보안 장과 함께 설명한다. 이용객 맞춤형 안내는 단순 화면 분기보다, 이용객 유형과 수속 조건에 맞는 안내 경로를 제공하는 형태가 더 적합하다.

3. 데이터 연계와 시스템 구성

3.1 데이터 소스

데이터 소스는 체크인, CUTE/CUPPS, 보안검색기록관리, iFIS, A-CDM, LiDAR 등 RFP에 명시된 후보군을 기준으로 정리한다. 공항별 실제 가용성은 다르므로 후보군과 실제 연계 확정값을 구분해야 한다. (DAR-001~DAR-006, EV-07, EV-13)

3.2 시스템 장비와 소프트웨어 구성 (ECR-001~ECR-003)

  • 기존 인프라 활용 및 설치환경 구성
  • 추가 인프라 구축 및 설치 요건
  • 데이터 연계 서비스를 위한 S/W 구성

이 3개 요구는 하나의 장에서 분리되지 않고, 아키텍처 그림과 함께 설명하는 것이 좋다. 특히 기존 인프라를 최대한 활용하면서도, 연계 안정성과 확장성을 동시에 확보하는 균형이 중요하다.

3.3 인터페이스 요구사항 (INR-001~INR-004)

  • 공통 사용자 인터페이스
  • 내부 시스템 연계
  • 외부 서비스/Open API 연계
  • 연계 데이터 정합성 확보

이 항목은 기능 장에서 표로 정리하고, Ⅴ장 품질 장에서 검증 체계를 붙여야 한다. 즉, 인터페이스는 설계와 검증이 한 몸으로 다뤄져야 한다.

4. 공항별 적용 논리

  • 김포(GMP): 국내선 실적을 무리하게 국제선으로 일반화하지 않고, 공항별 운영환경 차이와 검증 방식만 설명한다.
  • 김해(PUS): LiDAR가 존재하므로 기술 검증의 reference anchor로 사용한다.
  • 제주(CJU), 대구(TAE), 청주(CJJ): 실제 연계 가능 데이터원과 수동실측 기반 검증을 우선하며, 미확정 데이터는 보류로 둔다.

6. 시스템 아키텍처 서술 포인트

기술 및 기능 장에서는 단순히 “있다”를 나열하지 말고, 어떤 계층에서 무엇을 처리하는지 분명히 적는 것이 좋다. 예를 들어, 수집 계층에서는 원천 데이터와 메타데이터를, 처리 계층에서는 정제·보정·예측을, 서비스 계층에서는 API·대시보드·안내를, 운영 계층에서는 로그·모니터링·경보를 담당한다고 서술하면 구조적 이해가 쉬워진다.

7. 수속 단계별 상세 운영 시나리오

공항 운영에서는 같은 국제선이라도 시간대, 항공편 구성, 보안검색 혼잡도, 출국심사 처리량에 따라 체감 소요시간이 크게 달라진다. 따라서 본 서비스는 단순한 평균값 제시가 아니라, 단계별 상황을 시나리오로 분해해 보여주는 구조가 필요하다. 예를 들어 피크 시간대에는 보안검색 단계의 혼잡도가 상승하므로 대기 시간이 길어질 수 있고, 항공편 도착이 몰리는 구간에는 출국심사 단계의 처리 속도가 중요해진다. 제안서는 이러한 시나리오를 반영해 단계별 안내, 경보, 예외처리, 운영자 조치가 일관되게 이어지는 구조를 설명해야 한다.

8. 인터페이스와 오류 처리의 상세 원칙

연계 시스템에서 가장 중요한 것은 정상 상태보다 예외 상태를 어떻게 다루는가이다. API 응답은 성공 응답뿐 아니라 지연, 누락, 불일치, 권한 부족, 비식별 실패, 재시도 필요 상태를 함께 정의해야 한다. 운영자 대시보드도 정상값만 보여주는 화면이 아니라, 현재 어떤 데이터가 신뢰 가능한지, 어떤 데이터가 보류인지, 어떤 공항이 대체 로직으로 동작 중인지가 명확히 보여야 한다. 이와 같은 오류 처리 원칙은 시스템의 안정성을 설득하는 핵심 근거가 된다.

9. 요구사항별 기술 서술의 깊이

기능 장에서는 각 요구사항이 하나의 문단 이상으로 설명되어야 한다. 예를 들어 SFR-001은 단계 정의와 관리, SFR-002는 실시간 산출과 보정, SFR-003은 혼잡도 등급화, SFR-004는 예측과 검증, SFR-005는 API 노출, SFR-006은 운영자 화면, SFR-007은 리포트, SFR-008은 맞춤형 안내로 각각 나눠 적는다. 이렇게 써야 평가자가 요구사항과 문장을 즉시 연결할 수 있다. 또한 다이어그램은 기능 간 관계를 보여주는 데 중점을 두고, 화면 스케치는 운영 흐름을 이해시키는 정도로 제시하면 충분하다.
