Ⅳ. 기술 및 기능

1. 기술 구현의 전반 구조

이 장은 실제 시스템이 어떤 기능으로 구성되고, 어떤 데이터와 인터페이스를 통해 동작하는지 설명한다. 기능은 단순 목록이 아니라, 수속단계 정의 → 실시간 산출 → 혼잡도 판단 → 예측 → API/대시보드/리포트/맞춤형 안내로 이어지는 흐름으로 서술한다. 기능별 설명은 반드시 RFP 기능 요구사항과 연결되며, 새로 만든 주장이나 수치는 Stage 2.5 승인 없이 사용하지 않는다.

2. 기능별 상세 서술

2.1 국제선 수속단계 정의 및 관리 (SFR-001)

국제선 수속단계는 공항별 운영 환경에 따라 단계와 예외 규칙이 다르므로, 공통 단계 프레임과 공항별 단계 정의를 분리해 관리한다. 제안서는 표준 단계 정의, 예외 단계 등록, 단계별 관리항목, 변경 이력 관리 방식을 함께 제시해야 한다.

2.2 단계별 실시간 소요시간 산출 (SFR-002)

실시간 소요시간은 체크인, 보안검색, 출국심사, 탑승구 이동 등 단계별로 산출되며, 데이터 지연·누락·이상치에 대비한 전처리와 대체 산정 규칙이 필수다.

2.3 단계별 혼잡도 산출 및 등급화 (SFR-003)

혼잡도는 단순 체감치가 아니라 운영자가 이해할 수 있는 등급 체계로 정의해야 한다. 기준값, 경보 기준, 단계별 표시 방식, 운영자 화면 반영 방식을 함께 적는다.

2.4 AI 기반 소요시간 및 혼잡도 예측 (SFR-004)

예측 모델은 국제선 유형·환경별 데이터 테이블, 항공편 스케줄, 시간대, 요일, 예상 승객수, 기상, 공항별 운영특성을 반영해야 한다. 본문에서는 모델명보다 입력 특성, 검증 프레임, 오차 분석, 운영 적합성 평가에 초점을 둔다.

2.5 API 개발 및 서비스 노출 (SFR-005)

API는 내부 운영 시스템과 외부 서비스가 공유할 수 있도록 설계하되, 제공 가능 항목, 인증, 권한, 상태코드, 에러 응답을 먼저 정의한다.

2.6 운영자용 대시보드 (SFR-006)

운영자 화면은 실시간 상태, 예측, 경보, 정합성 이슈, 공항별 예외를 동시에 볼 수 있어야 한다.

2.7 통계 분석 및 리포트 (SFR-007)

리포트는 서비스 운영 현황, 오차 추이, 예외 발생률, 공항별 특이사항, 개선 과제를 정리하는 형태로 설계한다.

2.8 이용객 맞춤형 안내 (SFR-008)

이용객 조건에 따라 수속 소요시간과 혼잡도 안내를 개인화할 수 있어야 한다. 다만 개인정보 활용과 비식별 기준은 보안 장과 함께 설명한다.

3. 데이터 연계와 시스템 구성

3.1 데이터 소스

데이터 소스는 체크인, CUTE/CUPPS, 보안검색기록관리, iFIS, A-CDM, LiDAR 등 RFP에 명시된 후보군을 기준으로 정리한다. 공항별 실제 가용성은 다르므로 후보군과 실제 연계 확정값을 구분해야 한다. (DAR-001~DAR-006, EV-07, EV-13)

3.2 시스템 장비와 소프트웨어 구성 (ECR-001~ECR-003)

  • 기존 인프라 활용 및 설치환경 구성
  • 추가 인프라 구축 및 설치 요건
  • 데이터 연계 서비스를 위한 S/W 구성

이 3개 요구는 하나의 장에서 분리되지 않고, 아키텍처 그림과 함께 설명하는 것이 좋다.

3.3 인터페이스 요구사항 (INR-001~INR-004)

  • 공통 사용자 인터페이스
  • 내부 시스템 연계
  • 외부 서비스/Open API 연계
  • 연계 데이터 정합성 확보

이 항목은 기능 장에서 표로 정리하고, Ⅴ장 품질 장에서 검증 체계를 붙여야 한다.

4. 공항별 적용 논리

  • 김포(GMP): 국내선 실적을 무리하게 국제선으로 일반화하지 않고, 공항별 운영환경 차이와 검증 방식만 설명한다.
  • 김해(PUS): LiDAR가 존재하므로 기술 검증의 reference anchor로 사용한다.
  • 제주(CJU), 대구(TAE), 청주(CJJ): 실제 연계 가능 데이터원과 수동실측 기반 검증을 우선하며, 미확정 데이터는 보류로 둔다.
