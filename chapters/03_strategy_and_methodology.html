Ⅲ. 전략 및 방법론

1. 장의 역할

이 장은 제안서 전체의 논리 spine을 고정하는 장이다. 무엇을, 왜, 어떤 근거로, 어떤 범위에서 검증할 것인지 설명하고, 공항별 validation logic을 공통 방법론 아래에서 분기시키는 역할을 한다. Stage 1~2.5에서 고정한 요구사항 추적, evidence admissibility, 공통 spine 수렴 논리를 본문 문장으로 전환한다.

2. 방법론의 기본 구조

2.1 요구사항 분해

RFP 평가축을 기능, 성능, 시스템 구성, 인터페이스, 데이터, 테스트, 보안, 품질, 프로젝트 관리, 프로젝트 지원으로 분해한다. 이 분해는 01_RFP_Traceability_Matrix와 일치해야 하며, 각 세부 요구사항은 본문 어딘가에 반드시 대응되어야 한다. (CNR-001~CNR-004, SFR-001~SFR-008, PER-001~PER-002, ECR-001~ECR-003, INR-001~INR-004, DAR-001~DAR-006, TER-001~TER-003, SER-001~SER-004, QUR-001~QUR-003, PMR-001~PMR-004, PSR-001~PSR-002)

2.2 증거 수집과 판정

증거는 RFP 본문, 씽킹노트, RALPLAN 합의안, 그리고 Stage 1~2.5 산출물만 사용한다. claim은 source / owner / airport scope / comparator role / anonymity risk / page-budget impact를 기준으로 판정한다. 이때 공항별 데이터를 일반화하려는 유혹을 차단하고, evidence가 불충분하면 기본값은 보류 또는 quarantine이다. (EV-01~EV-18, CL-01~CL-12)

2.3 공통 수렴과 분기

공통 메시지 spine은 사업 필요성, RFP 정렬, 공항별 검증 철학, 데이터·보안·품질·테스트의 동시 설계를 포함한다. 그 위에 공항별 validation card를 얹어 각 공항의 차이를 설명한다. 따라서 본문은 공통화가 가능한 내용과 공항별로 분리해야 하는 내용을 명시적으로 분리한다.

3. RALPLAN 기반 결정 원칙

  • 공통 메시지는 먼저 세우되, 공항별 검증은 하위 구조로 둔다.
  • 김해 LiDAR는 source of evidence와 benchmark를 구분해서 쓴다.
  • 약한 근거는 fail-closed로 처리한다.
  • 본문과 발표자료는 동일 ledger를 공유한다.
  • Compliance와 Data/AI veto는 해소되기 전 무시할 수 없다.

4. 공항별 validation logic

| 공항 | 핵심 검증 방식 | 주의점 | 본문 표현 원칙 |
| --- | --- | --- | --- |
| 김포(GMP) | 수동실측 + 예측 비교 | 표본 편차 가능성 | 보수적 검증, 정량 과장 금지 |
| 김해(PUS) | 수동실측 + LiDAR + 티켓데이터 3중 비교 | LiDAR 커버리지와 단계 정합성 | anchor 공항으로만 사용 |
| 제주(CJU) | 수동실측 중심 + 계절성 고려 | 계절 변동, 샘플 편차 | 조건부/보수적 서술 |
| 대구(TAE) | 수동실측 + 보수적 예측 검증 | 데이터 밀도 부족 가능성 | 방법론 중심 서술 |
| 청주(CJJ) | 초기 검증 중심 | 외부 협의 범위 불확실성 | 제약 명시형 서술 |

5. 게이트 운영 방식

  • Stage 1: 요구사항 → 평가항목 → 섹션 → owner → 근거 매핑
  • Stage 2: proof architecture와 validation card 설계
  • Stage 2.5: claim admissibility gate
  • Stage 2.5b: 공통 spine convergence gate
  • Stage 3: 승인된 claim만 본문에 반영

6. 공통 spine 서술 방식

공통 spine은 세 가지 층으로 서술하는 것이 가장 안정적이다. 첫째, 사업 필요성은 국제선 수속의 변동성과 이용객 불확실성 해소라는 관점에서 설명한다. 둘째, 방법론은 공통 표준과 공항별 예외를 동시에 관리하는 구조로 설명한다. 셋째, 검증 체계는 공항별 데이터 가용성 차이를 인정한 상태에서 단계별 validation logic을 운영하는 방식으로 설명한다. 이 세 층은 이후 기술 장과 성능 장으로 그대로 연결된다.

7. 본문에 넣을 표현

  • “공통 메시지 spine을 먼저 두고, 공항별 validation은 하위 구조로 둔다.”
  • “김해공항 LiDAR는 source of evidence이자 제한적 benchmark로만 사용한다.”
  • “약한 근거는 공통 주장으로 승격하지 않고 fail-closed로 관리한다.”
